# Catalog schema

Reference for the read-only Mongo catalog backing the matcher. Generated by
running `node --env-file=.env scripts/explore-db.mjs` against the live Atlas
cluster. **The catalog is read-only — do not write to it, do not propose
schema changes.**

## Connection

| Setting     | Value                                         |
|-------------|-----------------------------------------------|
| Database    | `catalog`                                     |
| Collection  | `products`                                    |
| Document count | ~2,500                                     |
| Auth        | URI-embedded user/password (in `.env` `DB_URL`) |

The DB name is encoded in the URI path (`/catalog`). The collection name is
not in the URI; it is auto-detected by the script (it is the only
non-system collection) and is documented here.

## Indexes

Three indexes confirmed on `products`:

| Name                              | Key                                      | Notes                                                |
|-----------------------------------|------------------------------------------|------------------------------------------------------|
| `_id_`                            | `{ _id: 1 }`                             | Default Mongo primary key.                           |
| `title_text_description_text`     | `{ title: text, description: text }`     | Weighted text index — `title: 2`, `description: 1`. Default language `english`. |
| `category_1_type_1_price_1`       | `{ category: 1, type: 1, price: 1 }`     | Compound; supports filter narrowing then price sort. |

The text index is the foundation of retrieval (`$text` + `$meta:textScore`).
The compound index supports hard-filter narrowing on category/type and
range filtering on price.

There is **no** vector index, so semantic search must come from another
layer (LLM-driven query refinement and/or rerank).

## Document shape

Sampled five random documents; every field below is present on 5/5 with the
type listed. No nulls observed.

| Field         | Type      | Example                                              |
|---------------|-----------|------------------------------------------------------|
| `_id`         | ObjectId  | `6989d492d0ca8969ace9fe2b`                           |
| `title`       | string    | `"Modern Leather Dining Bench"`                      |
| `description` | string    | `"Natural modern dining bench made from premium leather. This benche piece is …"` |
| `category`    | string    | `"Benches"`                                          |
| `type`        | string    | `"Dining Bench"`                                     |
| `price`       | number    | `269.99`                                             |
| `width`       | number    | `135` (cm)                                           |
| `height`      | number    | `49` (cm)                                            |
| `depth`       | number    | `41` (cm)                                            |

There are **no** structured `style`, `material`, or `color` fields. Those
attributes appear baked into the leading sentence of `description`
(e.g. `"Natural modern … made from premium leather"`). Recovering them is
either an LLM extraction step or a lightweight regex pass against the
description prefix.

The TypeScript binding lives at `shared/src/catalog.ts` (`Product`).

## Cardinality of categorical fields

From the live DB:

- `category` — **15** distinct values: `Beds`, `Benches`, `Bookshelves`,
  `Cabinets`, `Chairs`, `Coffee Tables`, `Desks`, `Dressers`, `Lighting`,
  `Nightstands`, `Ottomans`, `Sofas`, `TV Stands`, `Tables`, `Wardrobes`.
- `type` — **62** distinct values, each scoped to a category
  (e.g. `Dining Bench`, `Floating TV Stand`, `L-Shaped Desk`,
  `Chesterfield Sofa`).
- `style`, `material`, `color` — **not stored as top-level fields**
  (`distinct()` returns `[]`).

## Implications for retrieval

1. Use `$text` against `{ title, description }` for the base query, sorted
   by `$meta:"textScore"`.
2. Hard-filter on `category` (and optionally `type`) using the compound
   index. Do this only when the vision extractor returns a high-confidence
   value from the closed `PRODUCT_CATEGORIES` set.
3. Price filtering can ride the same compound index; treat it as optional.
4. For style/material/color filtering, either extract from description on
   ingest-equivalent (e.g. precompute in the pipeline) or rely on the LLM
   rerank to handle those signals semantically.
